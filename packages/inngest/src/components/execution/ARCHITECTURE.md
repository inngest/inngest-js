# Execution architecture

## Lazy ops

Most opcodes without a local handler (`Sleep`, `WaitForEvent`, etc.) suspend the run. The SDK ships the opcode, the backend schedules the work, then the backend re-invokes the function with the result memoized in state. The opcode *is* the suspension.

Lazy ops (`StepOpCode.DeferAdd`) are fire-and-forget. User code keeps executing in the same run, so the op has no natural moment to ship. We resolve the user's `await` eagerly and park the op in `state.lazyOps` until the next wire message carries it (checkpoint or response).

`defer()` must work anywhere in a handler: top-level, inside `step.run()`, and in any combination. Routing through the core loop handles the top-level case fine but deadlocks inside `step.run()`, since the outer step already holds the loop. Buffering handles both cases uniformly and lets consecutive calls batch into a single ship for free.

The buffer drains onto whichever wire message comes next: an outbound checkpoint request (async-checkpointing mode) or the response the SDK returns to the executor (sync mode). Terminal paths bundle buffered ops alongside the terminal op via `attachLazyOps` (below), except the async-checkpointing rejection path, which attempts a final checkpoint first.

Lazy ops bypass `state.steps` and the step middleware pipeline entirely. They aren't user-addressable as steps (no handler, no memoization, no awaitable result), so step hooks like `transformStepInput` / `wrapStep` / `onStepStart` have nothing to do with them. `defer()` doesn't route through the engine's step handler at all: the engine's `registerLazyOp` hashes the ID and builds the op, then `LazyOps.register` decides pushed/duplicate/replay-skip; `priorDefers` (sent by the executor in the request payload) is injected into `LazyOps` at construction.

The `LazyOps` class (`lazyOps.ts`) buffers sync ops with methods: `push(op)` appends, `drain()` takes and clears, `restore(ops)` puts a failed drain back at the front, `register(op)` decides the op's fate (pushed/duplicate/replay-skip), and `abort(id, op)` handles abort idempotency and abortability checks. LazyOps owns buffer state and id-identity decisions (deduplication, replay filtering, abort idempotency/abortability); the engine owns op construction, hashing, logging, and shipping. The async `function-resolved` / `function-rejected` / `steps-found` handlers route their final response through `attachLazyOps`, which drains the buffer and bundles ops alongside the terminal `RunComplete` / `StepError` / `StepFailed` op (converting a terminal result into `steps-found` when ops are buffered). The remaining drain sites (`checkpoint()` and `maybeReturnNewSteps`) drain directly because they ship raw `OutgoingOp[]`, not an `ExecutionResult`. `checkpoint()` restores its drain on API failure so the ops ride the fallback response or a later message instead of being lost. Sync-mode terminal rejection still doesn't go through `attachLazyOps`; see the Todo in `defer.md`.

The synthetic rejection op's error carries a `RUN_REJECTION_MARKER` in its `data` field (the only error field the executor echoes verbatim) so the error-discovery replay short-circuits with the prior rejection instead of re-running user code (`memoizedRunRejection`). The marker — not the op's id — distinguishes it from a failed user step named `complete`, whose error must still be rethrown into the handler.

A `DeferAdd` and the `DeferAbort` targeting it must never ship in the same message — the executor processes a batch concurrently, so the pair can race. `LazyOps` enforces this as a buffer invariant (see its docblock), which holds wire-wide because defer ops only ever enter a batch via a single `drain()`; nothing may hold drained ops across an `await` and merge them into a later batch.
